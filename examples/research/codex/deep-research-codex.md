Ключевой вывод по твоей целевой зоне `Core -> Contract -> Module Implementation`: у codex-rs самые интересные идеи лежат не в UI и не в CLI, а в отдельных модулях поиска, policy-решений, patch-рантаймах, tool exposure, event-driven bridge к dynamic tools и в model/provider abstraction. По твоему пункту про repo understanding важно сказать прямо: в просмотренных публичных исходниках я **не подтвердил** отдельный subsystem для embeddings, AST/LSP, repo map или explainable “why this file entered context”. Подтверждены другие primitives: file-path search session, `list_dir`, grep-oriented handlers/tests, shell/unified exec, plugin/skill injections, compaction и patch/event-поверхность. Поэтому у этого агента “понимание репозитория” выглядит не как один умный indexer, а как композиция search/tools/context injections плюс compaction. Всё, что выше этого уровня, по публичным источникам **не подтверждено**. citeturn18view0turn19view0turn44view1turn42view6

## Context, Search и Tools

Перед карточками важно зафиксировать общий паттерн. В codex-rs поиск и контекст — это не одна связанная “ContextBuilder engine”, а набор отдельных механизмов: отдельный crate для file-path fuzzy search, tool-registry planning, deferred exposure для слишком больших наборов MCP tools, BM25-поиск по описаниям инструментов, explicit plugin mention injection и автоматическая compaction перед очередным model turn. Для твоей архитектуры это скорее аргумент **в пользу маленьких контрактов**, чем в пользу одного толстого context core. citeturn19view0turn31view1turn24view0turn44view1turn42view6

### Карточка метода

**Название метода:** Incremental fuzzy file-path search session. **Краткое описание:** отдельный crate запускает параллельный walker по корням репозитория и отдельно matcher на `nucleo`, пересчитывая top-N матчи по мере сканирования; есть exclude patterns, уважение к gitignore-семантике, optional hit indices для подсветки и snapshot-репортинг прогресса. **Где найдено:** `codex-rs/file-search/Cargo.toml`, `codex-rs/file-search/src/lib.rs`. **Ссылка на источник:** `file-search/src/lib.rs`, структуры `FileSearchOptions`, `FileSearchSession`, `walker_worker`, `matcher_worker`. **Подсистема агента:** Search / repo discovery. **Проблема, которую решает:** дешёвый и быстрый способ находить релевантные пути в большом workspace без embeddings и без полного rebuild индекса на каждый запрос. citeturn16view0turn20view0turn20view1turn20view2turn22view0turn22view2turn22view4

**Входные данные:** roots, query, `limit`, `exclude`, `threads`, `compute_indices`, `respect_gitignore`, cancel flag. **Внутренний алгоритм по шагам:** сначала строится override matcher для excludes; затем поднимаются два worker-а — walker и matcher; walker проходит hidden entries и symlinks, но приводит ignore semantics к git-поведению через `require_git(true)` и умеет отключать все ignore-правила целиком; найденные relative paths инжектятся в matcher; matcher периодически делает `tick`, берёт snapshot матчей, optionally считает indices, сохраняет `total_match_count`, `scanned_file_count`, `walk_complete` и отдаёт debounced updates. **Состояние, которое хранит:** текущий query, limit, flags, roots, shutdown/cancel bits, latest snapshot. **Выходные данные:** `FileSearchSnapshot` и финальный `FileSearchResults`. **Failure modes:** пустой roots → ошибка; пустой CLI query → просто `ls`; invalid exclude pattern → ошибка builder-а; cancel/shutdown прерывают session. citeturn20view0turn20view5turn22view0turn22view1turn22view2turn22view3turn22view5

**Зависимости от UI / модели / языка:** UI-зависимость низкая, только для подсветки индексов и прогресс-снапшотов; от модели метод не зависит; от языка программирования тоже не зависит. **Можно ли перенести в мой агент:** да, это один из лучших кандидатов. **В какой мой slot ложится:** прежде всего `SearchBackend`, опционально как источник для `ContextBuilder`. **Нужен ли новый contract:** только если ты хочешь не просто `search(query)->Vec<Path>`, а streaming snapshots; тогда полезен минимальный `SearchSession` contract. **Можно ли сделать dylib plugin / configured process/MCP tool:** да и так, и так; practically я бы делал dylib/process module, а не MCP. **Что должно остаться в core:** lifecycle, cancellation wiring, связь с task/session. **Что должно быть в plugin/module:** walker, ranking, snapshot streaming. **MVP-реализация:** один локальный path-fuzzy backend без content-search. **Config example:** `search.file_path = { backend = "nucleo", limit = 20, threads = 2, respect_gitignore = true }`. **Нужные DTO/events:** `SearchQuery`, `SearchSnapshot`, `SearchCompleted`. **Какие tests нужны:** ranking stability, gitignore scope, symlink behavior, cancellation, incremental update-before-complete. **Оценка:** impact — high; complexity — medium; boundary risk — low; priority — now. citeturn20view0turn20view2turn22view1turn22view3turn22view5

**Название метода:** BM25 tool search over deferred catalog. **Краткое описание:** deferred tools индексируются не семантическими embedding-ами, а через BM25 search engine over textual `search_text`; результаты потом bucket-limit-ятся и коалесцируются до `LoadableToolSpec`. **Где найдено:** `codex-rs/core/src/tools/handlers/tool_search.rs`, `codex-rs/core/src/tools/spec.rs`. **Ссылка на источник:** `ToolSearchHandler::new`, `SearchEngineBuilder::<usize>::with_documents(Language::English, documents).build()`, `search_result_entries`, `limit_results_by_bucket`, lazy creation handler-а в `tools/spec.rs`. **Подсистема агента:** Tool discovery / ToolRegistry. **Проблема, которую решает:** как найти нужный tool среди большого dynamic/deferred tool-set без показа всей вселенной инструментов модели сразу. citeturn23view5turn24view0turn24view1turn24view2turn24view3turn27view1turn27view2

## Permissions, Editing и Workflow

По safety/workflow части codex-rs особенно интересен тем, что здесь есть **два уровня orchestration**. Первый — policy-уровень для shell/MCP decisions; второй — generic tool orchestrator, который сводит runtime любого mutating tool к последовательности “approval → sandbox selection → attempt → retry/escalate”, причём approval path умеет hooks, guardian/user review и approval caching. Для твоей модульной архитектуры это сильный сигнал: `ApprovalPolicy` и `Workflow` лучше держать как отдельные contracts, а не как куски CLI/core. citeturn39view0turn44view6turn44view8

### Карточка метода

## Memory, Runtime, Model и Plugins

По runtime/event стороне я подтверждаю event-driven поверхность, а не полноценный “UI subsystem review”.
