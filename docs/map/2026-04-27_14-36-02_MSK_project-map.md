## Главная идея

Проект является модульным каркасом:

```text
Core -> Contract -> Module Implementation
```

Core не должен знать детали конкретного поиска, памяти, модели, tools, policy, patch algorithm или renderer. Новая функциональность должна проходить через существующий slot или через явно добавленный contract.

В v0 модульность означает не marketplace и не hot-reload, а выбор встроенной реализации через config:

```text
AppConfig -> BuiltinRegistry::from_config -> trait objects -> AgentRuntime
```

## Поток выполнения

Главная runtime-цепочка:

```text
CLI
-> AgentRuntime
-> BuiltinRegistry
-> RuntimeContext
-> SingleLoopWorkflow
-> ContextBuilder
-> ModelService
-> RequestShaper
-> ModelAdapter
-> ToolRegistry / ApprovalPolicy, если модель попросила tools
-> Renderer
```

Более подробно:

```text
task
-> Event::SessionStarted
-> Event::TaskReceived
-> ContextBuilder::build
-> Event::ContextBuilt
-> CanonicalModelRequest
-> ModelService::complete
-> RequestShaper::shape(request, ModelCapabilities)
-> ModelAdapter::complete
-> Event::ModelResponseReceived
-> если есть tool calls:
     PermissionMode gate
     ApprovalPolicy::evaluate в normal mode
     ApprovalTransport, если нужен approval
     ToolRegistry::get
     Tool::invoke
     Event::ToolFinished
     повторить model call
-> AgentOutput
-> Renderer::render
```

Лимит tool rounds в `SingleLoopWorkflow`: `4`.

Новый файл, который уже был в рабочем дереве:

Смысл текущего изменения:

- provider adapters (`openai`, `anthropic`, `fake`) теперь выглядят как `ModelAdapter`, а не прямой `ModelClient`;
- новый `ModelService` реализует `ModelClient`, оборачивает выбранный `ModelAdapter` и всегда прогоняет request через `RequestShaper`;
- `RequestShaper` начал смотреть на `ModelCapabilities`: убирает неподдерживаемые tools/cache/reasoning и ограничивает token limits;
- `BuiltinRegistry::from_config` теперь выбирает `ModelAdapter`, затем заворачивает его в `ModelService`;
- тесты расширены проверкой, что `ModelService` действительно shape-ит request перед adapter call;
- документация синхронизирована с новым model boundary.

Это архитектурно важное изменение: граница модели стала двухслойной.

```text
Workflow/Core -> ModelClient -> ModelService -> ModelAdapter -> provider API
```

## Tools и безопасность

Встроенные tools:

| Tool | Safety | Что делает |
|---|---|---|
| `read_file` | `ReadOnly` | Читает UTF-8 файл внутри workspace |
| `list_dir` | `ReadOnly` | Показывает файлы и директории внутри workspace |
| `search` | `ReadOnly` | Вызывает выбранный `SearchBackend` |
| `apply_patch` | `WritesFiles` | Применяет workspace-scoped patch через `PatchApplier` |
| `write_file` | `WritesFiles` | Пишет UTF-8 файл внутри workspace |
| `shell` | `RunsCommands` | Запускает команду в `cwd` |

Security-цепочка:

```text
ToolSpec.ToolSafety
-> PermissionMode
-> ApprovalPolicy
-> ApprovalTransport, если нужен вопрос пользователю
-> Tool workspace/path checks
```

`ToolRegistry`, `ApprovalPolicy` и `ToolSafety` нельзя обходить.

## Что защищают тесты

Главный файл: `tests/module_swap.rs`.

Он фиксирует:

- `search = null|rg` не требует изменения runtime;
- `memory = none|jsonl` не требует изменения runtime;
- `policy = allow_all|ask_write` не ломает read-only tool execution;
- `PermissionMode::Plan` и `PermissionMode::Auto` меняют видимость tools без изменения runtime;
- `ToolRegistry` запрещает duplicate names, хранит source и возвращает specs в стабильном порядке;
- `apply_patch` применяет internal patch format только внутри workspace;
- `write_file` не выходит за workspace через parent traversal или symlink;
- `FakeModelClient` работает через canonical request/response;
- `ModelService` применяет `RequestShaper` перед provider adapter;
- JSON config может выбрать Anthropic provider и custom local provider URL;
- statusline renderer валидирует components/position/frame;
- workspace path encoding стабилен.

Минимальная проверка для архитектурных правок:

```bash
cargo test
```

Если менялись только `.md` файлы:

```bash
git diff --check
```

Но `cargo test` все равно желателен, потому что документация здесь описывает фактические контракты.

## Planned vs implemented

Важно не смешивать:

- `README.md` и `docs/*` описывают фактическое состояние v0;
- `MODULAR_AGENT_SPEC_RU.md` описывает vision/spec/roadmap;
- если spec начинает описывать факт, нужно явно разделять `implemented` и `planned`.

Текущие ограничения v0:

- `ModuleManifest` существует как DTO, но не участвует в registry;
- MCP provider еще не реализован, хотя `ToolRegistry` уже хранит `ToolSource`;
- `MemoryStore::remember` есть, но активный workflow использует только `recall`;
- streaming enum есть, но OpenAI/Anthropic clients сейчас используют non-streaming `complete`;
- `PatchApplied` и `MemoryWritten` events существуют, но текущий workflow их не испускает;
- TUI/headless approval для tools, требующих approval, сейчас приводит к отказу.

## Быстрый чек-лист перед правкой

Перед изменением спросить:

```text
1. Это новый behavior или новая implementation существующего slot?
2. Есть ли уже contract в src/contracts?
3. Не протекает ли provider-specific тип за src/adapters или model shaping layer?
4. Не обходит ли код ToolRegistry, PermissionMode, ApprovalPolicy или ToolSafety?
5. Нужно ли добавить key в BuiltinRegistry::from_config?
6. Нужно ли обновить config example?
7. Какой тест доказывает заменяемость?
8. Какой doc теперь обязан измениться?
```
