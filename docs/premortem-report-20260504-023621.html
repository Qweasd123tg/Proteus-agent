Modular Agent failed six months from now

This report assumes the Rust-first modular coding-agent did not become a dependable tool or a credible plugin platform, then works backward from the failure.

Synthesis

Most Likely Failure

Platform work outruns coding-agent quality

The project keeps improving slots, ABI, plugins, and docs while repo understanding, edit/test loop, diff review, evals, and workflow ergonomics stay weaker than daily coding work requires.

Most Dangerous Failure

Trust breaks once

A workspace escape, secret leak, unsafe shell/network action, or plugin crash in a real repository would damage confidence more than a mediocre workflow would.

Hidden Assumption

Clean contracts will create a good agent

Contracts preserve boundaries. They do not automatically produce usefulness, safety, install reproducibility, protocol stability, or strong coding-loop behavior.

Revised Plan

  1. Freeze new slots for 4-6 weeks unless an end-to-end eval proves the need.
  2. Make one golden coding profile excellent: repo-aware context, patching, approved shell/tests, diff/result reporting, cancellation, and resume.
  3. Build the eval harness before adding more platform surface.
  4. Add strict plugin `doctor` checks for stale, incompatible, duplicate, missing, and globally shadowed plugins.
  5. Move security from plugin promises to central enforcement and malicious-plugin tests.
  6. Stabilize app-server with protocol/state-machine tests for approvals, cancel, reconnect, resume, and shutdown.
  7. Add documentation inventory checks that compare code, manifests, module ids, and documented plugin/config lists.

Pre-Launch Checklist

  • `doctor --strict` detects stale global plugins and contract mismatches.
  • Five coding evals pass with recorded tests, tool calls, approvals, files, duration, and failure reason.
  • Security tests cover traversal, symlinks, env secrets, network defaults, `allow_all`, and `auto` mode floors.
  • TUI/app-server tests cover long turns, approval queue, cancel, timeout, reconnect, and resume.
  • Docs inventory fails when plugin lists, config examples, or implemented/planned status drift.

Failure Modes

1. Architectural modularity became a tax

LikelihoodHighSeverityHigh

Story

Useful features needed contract work, ABI glue, manifests, config, docs, and replaceability tests before the core behavior existed. To move faster, exceptions leaked back into runtime/app-server/UI.

Underlying Assumption

Universal slots are always cheaper than targeted integration, even for small or unstable capabilities.

Warning Signs

Feature work starts with "add another slot"; core gets `if plugin_id == ...` or permanent temporary fallbacks.

2. Dylib plugins became brittle

LikelihoodMediumSeverityHigh

Story

Old and new plugins mixed in global `~/.agent/plugins`. `install.sh` overwrote libraries, ABI/layout mismatches produced random crashes, and recovery became "delete plugins and rebuild".

Underlying Assumption

`abi_stable` plus contract discipline is enough without version resolution, isolation, and reproducible installation.

Warning Signs

Runtime breaks only on machines with stale installed plugins; bug reports are solved by clearing the plugin directory.

3. Architecture outran coding quality

LikelihoodHighSeverityHigh

Story

The agent looked modular but still failed the common loop: find relevant files, edit, show diff, run tests, interpret failures, and iterate without hand-holding.

Underlying Assumption

Good modularity will naturally lead to strong coding-agent behavior.

Warning Signs

Users manually open diffs, run tests, and feed context while new plugin slots appear faster than eval quality improves.

4. Security and trust failed

LikelihoodMediumSeverityCritical

Story

A plugin or config bypassed workspace boundaries, read secrets, changed sensitive files, or made network/shell behavior hard to reason about. Users could no longer trust safe modes.

Underlying Assumption

Plugin-author discipline and policy config can replace centralized enforcement.

Warning Signs

Security invariants live in docs but not malicious-plugin tests; safe modes depend on policy behavior rather than impossibility.

5. App-server and TUI stayed demo-grade

LikelihoodMediumSeverityHigh

Story

Long coding turns exposed stuck spinners, approval races, weak cancel/interrupt, and inconsistent resume/reconnect state. Protocol DTOs changed without a stable state-machine contract.

Underlying Assumption

A thin transport layer can be gradually hardened later without blocking real workflows.

Warning Signs

Reports say cancel or approve affected the wrong step; protocol/session bugs grow faster than integration tests.

6. Documentation drifted from code

LikelihoodHighSeverityMedium

Story

README, modules, configuration, roadmap, and spec kept old plugin lists and mixed planned with implemented. Contributors followed an attractive but false architecture map.

Underlying Assumption

Docs will stay synchronized without verification and visible implemented/planned separation.

Warning Signs

Reviews say "docs say otherwise"; new contributors ask questions that docs should answer but no longer can.

7. Tests gave false confidence

LikelihoodHighSeverityHigh

Story

`module_swap.rs` and unit tests stayed green, but real installed-plugin workflows broke across model, context, tool, patch, policy, renderer, and UI boundaries.

Underlying Assumption

Contract tests imply practical correctness of installed-plugin workflows and coding-loop quality.

Warning Signs

No regular eval harness; user bugs require a hand-built combination of config, installed dylib, model, and workflow to reproduce.

8. Scope split the project

LikelihoodHighSeverityHigh

Story

The project tried to mature a local coding-agent and a universal agent platform at once. It ended with too much config/contract machinery and not enough polished "open repo, get useful result" flow.

Underlying Assumption

A small project can mature product UX and platform guarantees in parallel without narrowing scope.

Warning Signs

More documents and contracts appear while end-to-end scenarios do not become visibly faster, safer, or more stable.
